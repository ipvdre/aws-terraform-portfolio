### Traffic Flows

| Source | Destination | Path |
|---|---|---|
| Home PC | Instance1 (public) | VPN Tunnel → VPN Gateway → Public Route Table → Instance1 |
| Home PC | Instance2 (private) | VPN Tunnel → VPN Gateway → Private Route Table → Instance2 |
| Instance2 | Internet (yum update) | Private RT → NAT Gateway → IGW → Internet |
| Instance1 | Internet | Public RT → IGW → Internet |

### Security Design

- Instance1 lives in the public subnet — reachable from internet via IGW, reachable from home via VPN
- Instance2 lives in the private subnet — no direct internet exposure, outbound only via NAT
- Both instances locked to home CIDR at NACL and Security Group level
- Defense in depth: stateless NACLs at subnet level + stateful Security Groups at instance level
- PSKs auto-generated by AWS, never hardcoded, marked `sensitive = true` in outputs
- Sensitive inputs (home IP, network CIDR) sourced from HashiCorp Vault
- No credentials committed to Git

## Prerequisites

Before running Terraform, you need:

1. **AWS CLI** installed and configured with SSO
2. **Terraform >= 1.5.0** installed
3. **HashiCorp Vault** running locally with KV v2 engine enabled at `secret/`
4. **Your home public IP** — run `curl ifconfig.me` from your home network
5. **Your home subnet CIDR** — the network behind your router (e.g. `192.168.1.0/24`)

## Deployment

### Step 1 — Authenticate to AWS

```bash
aws sso login --sso-session my-sso-session
aws sts get-caller-identity   # Verify — should return your account ID
```

### Step 2 — Create your tfvars file

```bash
cp terraform.tfvars.example terraform.tfvars
```

Open `terraform.tfvars` and fill in your values:

```hcl
aws_region       = "us-east-1"
vpc_cidr_block   = "10.10.0.0/16"
subnet_cidr_a    = "10.10.10.0/24"
subnet_cidr_b    = "10.10.20.0/24"
instance_type    = "t3.micro"
home_public_ip   = "YOUR.HOME.PUBLIC.IP"
home_subnet_cidr = "192.168.1.0/24"

availability_zone_a = "us-east-1a"
availability_zone_b = "us-east-1b"
security_group_name = "security-group1"
instance1_name      = "Instance1"
instance2_name      = "Instance2"
vault_token         = "YOUR_VAULT_TOKEN"
```

> `terraform.tfvars` is in `.gitignore`. It will never be committed. Do not remove it from `.gitignore`.

### Step 3 — Initialize and Plan

```bash
terraform init
terraform plan
```

Review the plan output. Verify the resource count before applying.  
Expected: approximately 30 resources to create.

### Step 4 — Apply

```bash
terraform apply
```

Type `yes` to confirm. Apply takes 3–5 minutes.  
The VPN connection is the slowest resource — AWS provisions both tunnel endpoints on their infrastructure.

### Step 5 — Retrieve VPN Tunnel Information

After apply completes, retrieve tunnel IPs from standard output:

```bash
terraform output vpn_tunnel1_address
terraform output vpn_tunnel2_address
```

PSKs are marked `sensitive = true` and must be retrieved explicitly:

```bash
terraform output vpn_tunnel1_preshared_key
terraform output vpn_tunnel2_preshared_key
```

> These values never appear in plan or apply logs. Retrieve them only when needed.

### Step 6 — Configure EdgeRouter 4

Using the tunnel IPs and PSKs from Step 5, configure two IPsec peers on your EdgeRouter 4:

| Field | Tunnel 1 | Tunnel 2 |
|---|---|---|
| Peer | `<tunnel1_address>` | `<tunnel2_address>` |
| Local IP | Your ER4 WAN interface IP | Same |
| Encryption | AES-256 | AES-256 |
| Hash | SHA1 | SHA1 |
| DH Group | 14 | 14 |
| Pre-shared secret | `<tunnel1_psk>` | `<tunnel2_psk>` |
| Local subnet | `192.168.1.0/24` | `192.168.1.0/24` |
| Remote subnet | `10.10.0.0/16` | `10.10.0.0/16` |

> If your EdgeRouter 4 WAN interface is behind a modem performing NAT (double NAT), forward UDP 500 and UDP 4500 from the modem to the ER4 WAN interface IP. Use the ER4 LAN-side IP as the Local IP in the peer config, not the public IP.

### Step 7 — Verify

**Check tunnel status from AWS CLI:**
```bash
aws ec2 describe-vpn-connections \
  --vpn-connection-ids $(terraform output -raw vpn_connection_id) \
  --query "VpnConnections[0].VgwTelemetry"
```

Expected: `Status: UP` on both tunnel endpoints.

**Ping EC2 instances from home:**
```bash
ping $(terraform output -raw instance1_private_ip)
ping $(terraform output -raw instance2_private_ip)
```

**Verify Instance2 has internet access via NAT:**
```bash
# SSH into Instance2 via VPN, then:
curl --max-time 5 https://ifconfig.me
# Should return the NAT Gateway's Elastic IP
```

**Verify Instance1 has internet access via IGW:**
```bash
# SSH into Instance1 via VPN, then:
curl --max-time 5 https://ifconfig.me
# Should return the IGW's public IP
```

## Troubleshooting

| Symptom | Likely Cause | Fix |
|---|---|---|
| Tunnels stay DOWN after apply | Crypto mismatch (IKE version, hash, DH group) | Verify ER4 negotiates IKEv1/SHA1. Run packet capture on ER4 eth0 UDP/500 |
| IKE packets never reach ER4 | Double NAT — modem in front of ER4 | Forward UDP 500 and 4500 on modem to ER4 WAN IP |
| Tunnels UP but no ping to EC2 | Route propagation not applied | Verify `aws_vpn_gateway_route_propagation` exists for both route tables |
| Ping works to Instance1, not Instance2 | Private route table missing VPN route | Confirm propagation block targets `aws_route_table.private.id` |
| Instance2 cannot reach internet | NAT Gateway or route misconfigured | Verify private route table has `0.0.0.0/0 → nat_gateway_id` |
| `terraform plan` returns no config | Wrong directory | `pwd` to confirm you are inside the project folder |
| PSKs not visible in apply output | Expected — they are `sensitive = true` | Use `terraform output vpn_tunnel1_preshared_key` explicitly |

## Security Notes

- NACLs are stateless. Ephemeral port rules (1024-65535) are required for return traffic.
- Security Groups are stateful. Inbound allow is sufficient — return traffic is automatic.
- SHA1 is accepted by EdgeRouter 4's IKEv1 engine. Upgrade to SHA2-256 with DH Group 19 when hardware supports IKEv2 natively.
- PSKs are AWS-generated and live only in Terraform state. Never hardcode them.
- State file contains sensitive values in plaintext. `*.tfstate` is in `.gitignore`.
- Vault token is passed via `terraform.tfvars` (git-ignored), never committed to source control.
